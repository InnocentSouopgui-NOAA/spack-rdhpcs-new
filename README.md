# spack-self-contained

A self-contained, tiered software stack built with Spack, organized by how
often each piece of software gets refreshed, with older versions kept around
(isolated from each other) instead of being overwritten.

## The tiering model

Software is grouped into three update-cadence tiers, refreshed within a
single calendar year:

- **annual** — refreshed once a year. The foundation tier.
- **semiannual** (`H1`, `H2`) — refreshed twice a year. Built with `annual`
  as its upstream.
- **quarterly** (`Q1`–`Q4`) — refreshed four times a year. `Q1`/`Q2` build
  with `H1` as upstream, `Q3`/`Q4` with `H2` as upstream.

Each year starts a fresh grouping — `annual-2027` does **not** chain from
`annual-2026`. Nothing is retained indefinitely across years; within a year,
nothing is retained indefinitely either except by virtue of tiers not being
deleted (older quarters/semiannuals just sit there, installed and available,
until the year rolls over).

The year boundary is also a compatibility boundary: the control plane is
isolated per year (own templates, own environment specs, own Spack clone),
specifically so that 2027 can move to a Spack release that isn't backward
compatible with however 2026 is set up, without touching 2026 at all. See
"Control plane vs. install plane" below.

"Upstream" here is Spack's own chaining mechanism (`upstreams.yaml`): each
tier is a fully separate Spack instance (own config scope, own install
tree), and a downstream instance reuses whatever is already built in its
upstream instead of rebuilding it. Upstream resolution is transitive, so a
quarterly instance only needs to declare its semiannual as upstream — the
semiannual's own link to `annual` makes `annual`'s packages reachable too
without a second declaration.

## Control plane vs. install plane

Two directory trees, deliberately on separate disks — and the control side
is additionally partitioned per year:

- **`control/<year>/`** — everything hand-authored/version-controllable
  *for that year*: its own Spack clone (`spack/`), its own shared compiler
  scope (`common/config/compilers.yaml`), per-tier environment specs
  (`instances/<tier>/environment/spack.yaml`), *templates* for per-tier
  Spack config (`templates/`), and its own **meta** tree (`meta/`) — the
  Lmod meta-module templates, standing in for a clone of a separate
  meta-module repo (see below). Nothing in here is generated by Spack, and
  nothing in here hardcodes the install root. One year's `control/<year>/`
  shares nothing with another's — that's the point: `control/2027/spack`
  can be a completely different, incompatible Spack release from
  `control/2026/spack`, with its own config schema; `control/2027/meta`
  can just as freely be an incompatible redesign of how meta-modules are
  put together; and neither year is affected by the other.
- **`control/<year>/bootstrap.sh`** — also isolated per year, not a
  shared tool. It's simple text templating today, but there's nothing
  stopping 2027's version from growing differently (or incompatibly)
  from 2026's, same as `spack/` and `meta/`.
- **install root** — a single directory (path of your choosing, e.g. on a
  scratch disk) holding both Spack's own output (install trees, build
  stage dirs, caches, generated per-package Lmod modulefiles) *and* the
  rendered config/meta-modules/init-script produced by `bootstrap.sh`.
  Everything under it is either generated by Spack or generated by the
  bootstrap step — nothing here is hand-edited, so wiping a year's slice
  and re-bootstrapping + re-installing reconstructs it from
  `control/<year>/`.

The install root's location is never hardcoded in `control/` — templates
use the placeholder `@@SPACK_INSTALL_ROOT@@`, and
`control/<year>/bootstrap.sh <install-root>` substitutes it once, at
setup time, writing concrete files under that root (the year itself comes
from where the script lives, not from an argument). See "Bootstrapping a
year" below for why this replaced having everyone export an environment
variable by hand.

```
control/                              (backed-up disk, git-tracked)
  2026/
    bootstrap.sh                     (isolated per year, see above)
    spack/                           (Spack clone pinned for 2026)
    common/config/compilers.yaml     (shared across 2026's tiers only, see below)
    instances/<tier>/environment/spack.yaml        (hand-authored specs)
    templates/
      instances/<tier>/spack-config/{config.yaml, upstreams.yaml, modules.yaml}
    meta/
      templates/<tier>.lua           (stands in for a clone of a meta-module repo)
  2027/
    bootstrap.sh                     (its own copy, free to diverge)
    spack/                           (a different, possibly incompatible Spack clone)
    common/config/compilers.yaml
    instances/<tier>/environment/spack.yaml
    templates/instances/<tier>/spack-config/...
    meta/templates/<tier>.lua        (can be an incompatible redesign vs. 2026's)

<install-root>/                       (scratch disk, generated output, not git-tracked)
  2026/
    bin/init.sh                      (generated — source before running spack/module, see below)
    instances/<tier>/spack-config/{config.yaml, upstreams.yaml, modules.yaml}  (rendered)
    user-cache                       (this year's Spack bootstrap/misc cache)
    <tier>/opt                       (install tree)
    <tier>/stage                     (build scratch space)
    <tier>/cache                     (per-instance misc cache)
    <tier>/modules/lmod              (generated per-package modulefiles)
  2027/
    ...                              (same shape, entirely separate)
  modules/meta/2026/<tier>.lua       (rendered — stable path, see below)
  modules/meta/2027/<tier>.lua
  cache/source                       (shared download cache, across years too)
```

Two different output rules are at play, and it's worth being explicit
about why: `control/<year>/templates/` (Spack config) mirrors under
`<install-root>/<year>/`, namespaced per year so nothing collides. But
`control/<year>/meta/templates/` renders to a **stable** path —
`<install-root>/modules/meta/<year>/<tier>.lua` — that doesn't move
depending on which year you're bootstrapping, because Lmod needs
`module load 2026/Q1` to resolve under one `module use` directory that
never changes. If meta's output were nested under a per-year prefix like
the Spack config is, that mount point would have to change every year,
which defeats the point.

## Bootstrapping a year

The install root needs to end up in a lot of places: every tier's
`config.yaml`/`upstreams.yaml`/`modules.yaml`, and every meta-module's
MODULEPATH entries. Rather than carrying it around as an environment
variable that every install command *and every user's shell* would need
set, `control/<year>/bootstrap.sh` renders it in once, at setup time:

```
control/2026/bootstrap.sh /scratch/spack-install
```

This reads `control/2026/templates/` and `control/2026/meta/templates/`,
replaces `@@SPACK_INSTALL_ROOT@@` with the literal path given, and writes:
- Spack config to `/scratch/spack-install/2026/instances/<tier>/spack-config/...`
- meta-modules to `/scratch/spack-install/modules/meta/2026/<tier>.lua`
- an init script to `/scratch/spack-install/2026/bin/init.sh` (see
  "Self-containment" below)

It only reads `control/2026/` and writes under the given install root —
it doesn't touch `$HOME`, `/etc`, or anything else system-wide, and (for
now — see "not yet built" below) it doesn't invoke `spack`, clone
anything, or install any software.

Run it once for initial setup, and again any time you edit that year's
templates or move to a different scratch disk — it's idempotent, just
overwrites that year's rendered output, never another year's.

## Self-containment: nothing goes in `$HOME`

By default Spack reads/writes user-level state in `~/.spack` (config
overrides, caches, its bootstrap store for things like `clingo`). To keep
this stack fully self-contained, every `spack`/`module` invocation against
a year's instances needs `SPACK_DISABLE_LOCAL_CONFIG=1` and
`SPACK_USER_CACHE_PATH` set. Rather than exporting those by hand,
`bootstrap.sh` writes them into a per-year init script — **source it**
before running anything else:

```
source /scratch/spack-install/2026/bin/init.sh
```

```
# contents of the generated init.sh
export SPACK_DISABLE_LOCAL_CONFIG=1
export SPACK_USER_CACHE_PATH="/scratch/spack-install/2026/user-cache"
```

`SPACK_DISABLE_LOCAL_CONFIG` stops Spack from reading (or writing to)
`~/.spack` and `/etc/spack` — only Spack's built-in defaults plus whatever
config scopes are explicitly passed via `-C` apply.
`SPACK_USER_CACHE_PATH` redirects Spack's own cache/bootstrap-store
directory (used regardless of the config-scope setting above) onto the
scratch disk instead of `~/.spack`. It's namespaced per year
(`<install-root>/<year>/user-cache`), not shared — since a different
year's Spack version could bootstrap incompatible tooling (`clingo`,
etc.) into it, sharing would defeat the same isolation the rest of this
design is built around.

This init script is unrelated to end users loading modules — nobody
loading `2026/Q1` needs to source anything; it only matters for whoever
runs `spack` commands (installing, concretizing, refreshing modules).

Because local config is disabled, compiler definitions can no longer come
from `~/.spack/compilers.yaml` — they live in `control/<year>/common/
config/compilers.yaml` instead, a scope shared by every tier *within that
year* (see install instructions below). Each year gets its own, since a
different Spack version may detect or describe compilers differently.

The Spack tool itself (a clone of github.com/spack/spack) lives at
`control/<year>/spack/` — not a system-wide or home-directory install, and
not shared across years — so each year's control-plane slice, including
the exact Spack version it was built against, can be archived, upgraded,
or discarded independently of every other year and of the install root.

## Modules

Lmod is the module system. Each Spack instance generates its own modulefile
tree under `<install-root>/<year>/<tier>/modules/lmod` — these are never
merged or copied between tiers, and never across years either.

On top of that, the rendered **meta-modules**
(`<install-root>/modules/meta/<year>/<tier>.lua`, rendered from
`control/<year>/meta/templates/`) are the user-facing switches. Loading
one prepends MODULEPATH with that tier's own module root *and* every tier
upstream of it (most specific first), so loading `2026/Q1` exposes Q1, H1,
and annual's packages all at once. There's no conflict enforcement
(`family()`/`conflict()`) — loading multiple tiers or years side by side
is allowed and unmanaged by design.

`control/<year>/meta/` is deliberately kept separate from
`control/<year>/templates/` (the Spack config), even though today both are
just plain files with no real git behind them — it's meant to read as "the
meta-module tooling for this year," the same role `spack/` plays for "the
Spack version for this year." If meta-module generation ever grows into
its own actual repository, `control/<year>/meta/` is where that clone
would go, and its internal layout is free to change from year to year
since nothing outside `bootstrap.sh` depends on it beyond the one
`meta/templates/<tier>.lua` convention.

## Current contents (2026)

| Tier   | Upstream | Specs |
|--------|----------|-------|
| annual | —        | cmake, ghostscript, doxygen, universal-ctags |
| H1     | annual   | git-lfs, openmpi |
| Q1     | H1       | pmix, ucx, hwloc, intel-oneapi-compilers, intel-oneapi-mpi, intel-oneapi-mkl, hdf5+mpi ^intel-oneapi-mpi, netcdf-c+mpi ^intel-oneapi-mpi, netcdf-fortran ^intel-oneapi-mpi |

`hdf5`, `netcdf-c`, and `netcdf-fortran` are pinned to `intel-oneapi-mpi` as
their MPI provider.

`H2`, `Q2`–`Q4`, and future years are not yet scaffolded. A new tier within
2026 means copying the pattern under `control/2026/templates/` (new
`upstreams.yaml` pointing at the right semiannual/annual, new
`environment/spack.yaml` spec list) and `control/2026/meta/templates/`
(new meta-module chaining its own root plus its ancestors'), then
re-running `control/2026/bootstrap.sh <install-root>`. A new year
means creating `control/2027/` from scratch (its own `bootstrap.sh`,
`spack/` clone, `common/config/`, `instances/`/`templates/`/`meta/`) —
nothing from `control/2026/` is reused, by design.

## Installing

Not automated beyond `bootstrap.sh` — run the rest yourself. Full
step-by-step runbook: **[`control/2026/INSTALL.md`](control/2026/INSTALL.md)**
(clone Spack, bootstrap, source the init script, register compilers,
concretize as a smoke test, install in dependency order, generate Lmod
modules, wire up `module use`, verify upstream reuse). Each year gets its
own `INSTALL.md` alongside its own `bootstrap.sh` — 2027's may read
differently if the process changes along with the Spack/meta version for
that year.

## Not yet built

- Making the latest quarter available by default (without an explicit
  `module load`) — e.g. a `current`-style symlink or an always-on
  MODULEPATH entry.
- `bootstrap.sh` only renders templates and the init script today. It
  doesn't yet clone `spack/`/`meta/` for a new year, run
  `spack compiler find`, or otherwise scaffold a year end-to-end — those
  steps are still manual, per `control/2026/INSTALL.md`. The name leaves
  room for it to grow into that; ask when you want it.
- `control/<year>/meta/` is plain files today, not an actual git clone of
  anything — it's structured so that slotting in a real meta-module repo
  later is a drop-in change, not a redesign.
